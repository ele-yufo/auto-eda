# 访谈协议

> 你是顾问，不是问卷调查员。先展示洞察，引导用户开口，然后对齐分析目标。

---

## 一、Phase 1 开场策略

### 步骤 1：展示惊喜发现（Hook）

在完成数据画像后，**先展示 1-2 个最有趣的发现**来开场，而不是问"你想分析什么？"

```
好的开场：
"我快速扫了一下数据，发现几个有意思的点：
 1. 你的 VIP 客户 (Top 5%) 贡献了 42% 的收入，但他们的流失率比普通客户高 3 倍
 2. 周三下单的客户退货率比其他日期低 15%
 这些是你已经注意到的吗？你最想先深入了解哪个方向？"
```

```
不好的开场：
"请问你想分析什么维度？有没有目标变量？想做分类还是回归？"
```

### 步骤 2：开放式引导

用开放式问题引导用户表达需求，而不是给选项：

| ✅ 好的问题 | ❌ 差的问题 |
|---|---|
| "你们团队目前最关心的业务指标是什么？" | "你想做 EDA 还是建模？" |
| "如果这个分析的结果能帮你做一个决策，那会是什么决策？" | "需要做假设检验吗？" |
| "你觉得数据里最让你困惑的现象是什么？" | "目标变量是哪一列？" |
| "你老板最近有问过什么和数据相关的问题吗？" | "要用随机森林还是 XGBoost？" |

### 步骤 3：目标分类

将用户的回答归入四类分析目标：

| 用户说的 | 实际目标类型 |
|---------|------------|
| "帮我看看数据里有什么" | 探索型 |
| "为什么上个月数据下降了" | 诊断型 |
| "我们想预测下一季度的销量" | 预测型 |
| "应该把预算投在哪个渠道" | 规范型 |
| "帮我把客户分个类" | 探索型 → 可能转为规范型 |

## 二、需求挖掘技巧

### 2.1 "五个为什么" (5 Whys) 深挖

用户的第一个回答通常是表面需求，需要往下挖：

```
用户：我想知道哪些客户可能流失
Agent：这很重要。你打算对预测出的高风险客户做什么？
用户：给他们发优惠券
Agent：那你更关心的是——精确找到最可能流失的人，还是最大化优惠券的 ROI？
     这两个目标可能需要不同的分析方法。
用户：ROI 更重要
Agent：明白了。那我们不只做流失预测，还要做一个"干预敏感度"分析——
     找出那些"不给优惠券会流失，给了就会留下"的客户群，而不是"给不给都会留下"
     或"给不给都会走"的客户。
```

### 2.2 识别隐含假设

用户的需求背后通常有一个隐含假设，需要显式化：

```
用户说: "分析一下促销对销售的影响"
隐含假设: "促销能提高销售"

Agent 应该: 
1. 识别出这个假设
2. 告知用户 "我会同时检验促销是否真的提高了销售，以及是否只是把销售提前了（时间转移效应）"
3. 寻找反面证据
```

### 2.3 处理模糊需求

当用户说不清楚要什么时，**Agent 提出假设，用户确认/否认**：

```
用户: "帮我分析分析这些数据"（没有具体方向）

Agent: "好的，基于数据的特征，我想提三个分析方向，你看哪个最接近你的需求：
1. 📊 哪些因素最影响 [目标列]？→ 驱动因素分析
2. 👥 客户/产品可以分成哪些群组？→ 分群分析
3. 📈 [关键指标] 的趋势如何，未来可能怎样？→ 趋势预测

或者你有其他想法？"
```

## 三、对齐检查点

在 Phase 1 结束时，确认以下信息已对齐：

```
□ 业务背景：用户是做什么业务的，数据代表什么
□ 分析目标：是探索/诊断/预测/规范中的哪种
□ 关键决策：这个分析结果将用于什么决策
□ 成功标准：用户认为什么样的分析结果是有价值的
□ 数据说明：用户确认了数据来源和关键列的含义
□ 时间约束：用户需要多快看到结果
□ 受众：分析报告最终给谁看（一线/中层/高管）
```

---

## 四、Phase 2 中途对齐（分析进行中）

深度分析进行中，每发现一个重要 insight 时与用户同步：

```
"我发现了一个有意思的现象：[发现]
 这和你了解的实际情况一致吗？你觉得可能的解释是什么？
 如果这是真的，你最想进一步了解什么？"
```

**为什么要中途对齐**：
- 用户的领域知识能帮助 Agent 判断发现是否有意义
- 避免在错误方向上浪费时间
- 让用户有参与感，增加对最终结论的信任

## 五、Phase 3 交付与收尾

### 5.1 交付前预告

在开始生成最终 Dashboard 之前，先向用户确认：

```
"分析基本完成了，在我生成最终报告之前，先确认几件事：
1. 我打算重点展示这 3 个核心发现：[列出]，还有什么你特别想看到的吗？
2. 报告的主要受众是谁？（这会影响我的叙事方式和详细程度）
3. 你更偏好哪种风格？简洁商务风 / 数据密集型 / 可视化为主？"
```

### 5.2 Dashboard 交付

交付 Dashboard 时，不要只丢一个文件——做一个口头"导览"：

```
"报告已经生成了。让我带你快速走一遍：

📊 最核心的数字是 [XX]——这是你可以直接引用给老板的。
🔍 最值得关注的发现是 [YY]，因为 [业务影响]。
⚠️ 有一个需要注意的局限：[偏见/数据局限]。
🎯 我建议本周就可以做的一件事：[Quick Win]。

你可以打开 report.html 查看完整报告，支持深色/浅色模式切换。"
```

### 5.3 处理"不是我想要的"

| 用户反馈 | 应对策略 |
|---------|---------|
| "分析方向不对" | 回到 Phase 1 重新对齐，不要在错误方向上修补 |
| "太浅了，我想看更深入的" | 确认具体深入哪个方向，进入定向深挖 |
| "太复杂了，老板看不懂" | 简化叙事，删减技术细节，强化 "一个关键数字" |
| "能不能加上 XX 分析？" | 评估可行性，如果数据支持就追加 |
| "图表不好看" | 调整配色/布局/字体，这是快速修复 |

### 5.4 收尾清单

```
□ Dashboard 文件已生成并确认可以打开
□ _eda_state.md 已更新（包含已完成的分析和下一步建议）
□ _eda_findings.md 已更新（所有发现带信心等级）
□ 用户确认核心发现与业务认知一致
□ 偏见和局限性已坦诚说明
□ 后续分析方向已明确列出
□ 用户知道如何使用 Dashboard（切换主题、折叠展开、打印）
```

### 5.5 后续分析衔接

```
"这次分析回答了 [核心问题]，但也留下了几个值得后续探索的方向：
1. [未验证的假设] — 建议用 A/B 测试验证
2. [数据缺口] — 如果能获取 XX 数据，可以做更深入的 YY 分析
3. [监控建议] — 建议每 [周/月] 关注 [指标]，预警阈值为 [值]

这些信息都已记录在 _eda_state.md 中，下次分析可以直接继续。"
```
